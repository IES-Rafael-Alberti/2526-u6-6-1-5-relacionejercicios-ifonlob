II.c) ¿Existe otra forma de implementarlo que no sea herencia? (composición, interfaces, delegación, etc.)

RESPUESTA:

**Primera propuesta**:

Esta primera propuesta en vez de hacer uso de la herencia, cada una de las subclases podrían implementar una interfaz `HaceSonido` que defina el contrato del método `hacerSonido()`, como se muestra a continuación:

```kotlin
interface HaceSonido {
    fun hacerSonido()
}

class Perro(val nombre: String) : HaceSonido {
    override fun hacerSonido() = println("$nombre ladra: ¡Guau guau!")
}

class Gato(val nombre: String) : HaceSonido {
    override fun hacerSonido() = println("$nombre maúlla: ¡Miau!")
}

fun hacerSonidoAnimales(animales: List<HaceSonido>) {
    animales.forEach { it.hacerSonido() }
}
```

**Segunda propuesta**:

En esta segunda propuesta se podría llevar a cabo a través de composición en donde solo exista una clase `Empleado` que contenga todas las propiedades y todos los métodos tanto de la superclase como de las subclases,
pero que dependiendo de los valores de las propiedades específicas de las subclases se muestre un resultado u otro, como se puede apreciar a continuación:

```kotlin
class Empleado(
    val nombre: String,
    val apellido: String,
    val salarioBase: Double,
    val bonus: Double? = null,
    val horasTeletrabajo: Int? = null
) {
    fun trabajar() = println("$nombre $apellido está trabajando.")

    fun calcularSalario(): Double = salarioBase + (bonus ?: 0.0)

    fun solicitarVacaciones(dias: Int) {
        bonus?.let { println("$nombre solicita $dias vacaciones.") }
            ?: println("$nombre no tiene bonus/vacaciones.")
    }

    fun conectarVPN() {
        horasTeletrabajo?.let { println("$nombre conecta VPN ($it horas).") }
            ?: println("$nombre no es remoto.")
    }
}
```

II.d) ¿Qué ventajas aporta esta implementación con respecto a la alternativa?

RESPUESTA:
Primera propuesta:

Respecto a la primera propuesta considero que la versión con herencia es superior a usar únicamente la interfaz `HaceSonido`, 
ya que además de obligar a implementar `hacerSonido()` , la superclase Animal proporciona métodos compartidos como `comer()` y `moverse()`
que todos los animales tienen en común, evitando duplicar código en Perro y Gato. Esto modela mejor la relación "es un Animal" de forma natural y facilita el polimorfismo con un tipo común, mientras que con interfaces habría que repetir más lógica básica.


Segunda propuesta:

Para la segunda propuesta, la herencia desde Empleado ofrece ventajas claras sobre la composición con propiedades nullable que propuse, principalmente porque proporciona seguridad de tipos en tiempo de compilación , haciendo el código más legible y directo. 
Además, cada subclase sabe exactamente qué métodos tiene sin necesidad de revisar nulos, lo que simplifica añadir nuevos tipos de empleados sin tocar la base.
